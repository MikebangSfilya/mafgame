# Implementation Plan

## Решение

Первый прототип делается как легкий CLI-проект в:

```text
/home/sfilya/mafgame
```

Vault остается местом для дизайна, roadmap, задач, отчетов и решений.
Код прототипа, тесты и будущие файлы движка живут вне vault.

## Почему CLI сначала

- Можно работать короткими сессиями по 1-2 часа.
- Каждый шаг должен оставлять запускаемый результат.
- Главная проверка сейчас - интересен ли дневной цикл, приказы и отчеты.
- UI, Godot, карта и визуальные ассеты не должны мешать проверке ядра.

## Технологический порядок

1. Python CLI-прототип.
2. 20-дневный плейтест на 4 районах и 2 бригадах.
3. Решение: чинить ядро, расширять механику или переносить в Godot.
4. Godot только после того, как текстовый цикл дает интересные решения.

## Рабочий ритм

Одна сессия разработки должна закрывать одну маленькую задачу:

- добавить один экран вывода;
- добавить один приказ;
- добавить один тип отчета;
- поправить один параметр баланса;
- покрыть один сценарий проверки.

Нельзя оставлять прототип в состоянии, где его нельзя запустить.

## Связь с roadmap

Реализация идет по issue-порядку:

1. `ISS-20260527-001` - дневной цикл.
2. `ISS-20260527-002` - карта, банды и бригады.
3. `ISS-20260527-003` - 5 приказов и исходы.
4. `ISS-20260527-004` - оценки риска и отчеты.
5. `ISS-20260527-005` - враги, полиция, победа и поражение.
6. `ISS-20260527-006` - 20-дневный плейтест и баланс.

## Что не делать до проверки ядра

- Не начинать с Godot-сцен.
- Не делать визуальную карту.
- Не проектировать универсальную архитектуру.
- Не добавлять большой город.
- Не добавлять pathfinding, инвентарь, анимации или тактические бои.
- Не расширять карту, пока 4 района не дают интересный цикл.

## Минимальная структура CLI-проекта

```text
mafgame/
  README.md
  src/
    main.py
    model.py
    game.py
    orders.py
    reports.py
  data/
    initial_state.json
```

Структура может меняться, если это помогает быстрее проверить игровой цикл.
